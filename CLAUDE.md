# Global Claude Code Instructions

## About me

- Operating system: Linux
- Editor: Emacs (never suggest nano, vim, or other editors)
- Shell: bash
- SSH keys: `~/.ssh/` and via 1Password SSH agent
- Git remotes: accessed over SSH; multiple servers
- Primary languages: Python 3, Bash scripting

## General behavior

- Be concise. No unnecessary preamble or filler.
- Never explain what you are about to do — just do it.
- Prefer explicit over implicit.
- Always show me diffs or file changes before applying them.
- When in doubt about intent, ask before acting.
- Do not auto-accept or auto-run anything that modifies git history, deletes files, or touches credentials.

## Code style

- Write clean, readable code that a human will maintain.
- No AI-sounding comments ("This function handles...", "Here we...").
- No `// Generated by Claude` or similar attribution in code or comments.
- Match the existing style of the project being worked on.
- Prefer simple, obvious solutions over clever ones.
- Follow best practice coding styles and ensure that all code generated or changed will pass most common static code analys tools

## Git behavior

- Never commit without explicit confirmation from me (the pre-commit hook enforces this, but also ask).
- Never push without explicit instruction.
- Never amend or rebase without explicit instruction.
- Commit messages: short imperative first line, blank line, then detail if needed. No AI mentions.

## SSH and credentials

- SSH keys are in `~/.ssh/` or managed by 1Password SSH agent.
- Never print, log, or include key material anywhere.
- Never store credentials in code or config files.
- If a remote operation fails authentication, report the error and stop — do not attempt workarounds.

## Project onboarding — mandatory first step

**At the start of every session in a project directory, before doing anything else:**

1. Check whether a project-specific `CLAUDE.md` or `.claude/` directory exists.

2. Ask me:
   > "Is it OK to leave traces of Claude Code usage in this project?
   > (CLAUDE.md, .claude/, commit attribution, AI-related comments)"

3. Based on my answer, set one of two modes for this session:

### Mode: VISIBLE
Claude Code usage is acceptable in this project.
- May create/use `CLAUDE.md` and `.claude/` in the project
- Attribution stays disabled (global default)
- If no project `CLAUDE.md` exists, offer to create one using the appropriate template from `~/.claude/templates/`
- Verify `.claude/settings.local.json` is gitignored

### Mode: STEALTH
No traces of Claude Code usage in this project.
- Do not create `CLAUDE.md`, `.claude/`, or `.mcp.json` in the project
- Do not mention AI assistance in any comments, commit messages, or documentation
- Do not add Claude attribution to commits (already disabled globally, but double-check)
- Suggest adding `.claude/` to `.gitignore` if it does not already appear there
- Remind me of stealth mode at the start of each subsequent session in this project

4. **Re-ask this question at the start of every new session** in this project, regardless of what was decided before. Modes are not persisted — always confirm.

## Pre-commit check

A hook at `~/.claude/hooks/pre-commit-check.sh` intercepts every `git commit`.
It scans for:
- Staged `.claude/`, `CLAUDE.md`, `.mcp.json`, `settings.local.json` files
- AI-related keywords in diff content (`claude`, `AI`, `LLM`, `generated`, `copilot`, `co-authored-by`)
- AI-related keywords in the commit message
- Non-empty attribution settings

If anything suspicious is found, the commit is blocked and you must resolve each issue.
If everything is clean, you will be shown a summary and asked for explicit confirmation.

**Never bypass or disable this hook without telling me explicitly.**

## Yocto / embedded Linux projects

I work extensively with Yocto/OpenEmbedded. General awareness:
- Projects use `source <path>/oe-init-build-env <builddir>` to set up the environment
- Layer structure, build directories, machines, and distros vary significantly per project
- `bitbake`, `devtool`, `recipetool` are the primary tools
- `local.conf` and `bblayers.conf` are in the build directory
- Do not modify `local.conf` or `bblayers.conf` without showing me the diff first
- Do not run `bitbake` commands without explicit instruction

For each Yocto project, a project-specific `CLAUDE.md` should document the exact structure. Use the template at `~/.claude/templates/yocto/CLAUDE.md` as a starting point.
